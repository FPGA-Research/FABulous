# Routing Demand Evaluator

okay so the goal for that pass (routing_demand_evaluator) is:
- That pass should be more or less self-contained.
- We have options to give it the tile dir, or the individual files.
- We will use the PathFinder router similar what nextpnr router2 is using.
  Here we expose some options from that algorithm but we set useful defaults.
- Then we expose what demands should be tested, we will have a default set of demands
  but the user can choose also waht to use and waht not. We should think of good
  demands that have benefits we also should intermix that with random paths that
  can also be larger but in general we also want to test larger paths. Then we
  expose how many demand iterations the user want to test and an option of
  how many of then should be random. We must of course consider that the
  switch matrix not only has local wires also wires from outside and
  long distance wires. In general some useful demands are:
  The most critical hard demands are BEL input reachability, BEL output escape,
  straight/turn routing, local/neighbor feedback, and carry/direct-chain routes,
  can BELs drive the same input net.
  But maybe we should also consider:
    1. BEL output escape
       LUT/FF/BEL output -> N/E/S/W routing wires

    2. BEL input reachability
       N/E/S/W routing wires -> LUT/FF/BEL inputs

    3. Same-tile local feedback
       LUT/FF output -> LUT input in the same tile

    4. Neighbor feedback
       tile(x,y) BEL output -> tile(x+1,y), tile(x-1,y), tile(x,y+1), tile(x,y-1) BEL input

    5. Straight wire continuation
       N -> S, S -> N, E -> W, W -> E for same or compatible tracks

    6. Left/right turns
       N -> E/W, S -> E/W, E -> N/S, W -> N/S

    7. Short-to-long wire access
       local/single wire -> double/quad/hex wire

    8. Long-to-short wire exit
       double/quad/hex wire -> local/single wire -> BEL input

    9. Multi-hop distance routes
       BEL output -> BEL input at Manhattan distance 1, 2, 4, 8

    10. Fanout routes
        one BEL output -> 2, 4, 8 nearby BEL inputs

    11. Control-net routes
        one source -> many FF clock-enable/reset/select-like inputs

    12. Carry/direct-chain routes
        carry out -> neighboring carry in

    13. DSP/RAM access routes
        routing wires -> DSP/RAM inputs and DSP/RAM outputs -> routing wires

    14. IO access routes
        IO input -> fabric BEL input and BEL output -> IO output

    15. Random seeded routes
        random source/sink pairs with fixed seed, split by local/medium/long distance
- Then we can do optimization. We just have an parameter opt, noting more.
  Here we can optimize the the switch-matrix by itertively removing wires
  and check the demand. Here we want to expose opt_* parameters that will then
  be applied when opt=True. We have here parameters like how many demands are
  we allow to fail, how much iterations, maybe what demand category should fail
  waht number. then we have a number of how much percent we want to reduce the
  pips, but demand boundary is priority.
- Then how to do optimization: A scalable way to estimate PIP importance
  is Monte Carlo ablation: repeatedly remove random PIP batches, run the
  demand tester, measure demand loss, and estimate each PIP’s importance
  from how much worse results are when that PIP is removed versus
  when it is kept. This gives a statistical ranking before actual pruning.
  Remove only small batches of high-removal-score PIPs, rerun the demand tester
  after each batch, and keep the removal only if all hard demands still pass and
  the soft/random demand failure rate stays below the user threshold. After each
  accepted batch, recompute importance, because backup PIPs become more important
  once other PIPs are removed.
  Mabe that:  remove_score(PIP) = cost_saving(PIP) / (importance(PIP) + epsilon).
  But:
  A batch may contain 1 critical PIP + 99 useless PIPs.
  The demand drops a lot.
  Naively, all 100 PIPs look bad.
  So that why we could fit a simple regression:
  loss_k ≈ β0 + β1*x1 + β2*x2 + ... + βn*xn
  where:
  xi = 1 if PIP i was removed in experiment k
  xi = 0 otherwise
  So the flow could be:
  1. Start from user switch-matrix list.
  2. Generate random PIP-removal batches.
     Use different batch sizes:
       small: 1–2%
       medium: 5%
       large: 10–20%

  3. For each batch:
     remove PIPs
     run demand tester
     compute weighted loss
     restore PIPs

  4. Estimate PIP importance:
     importance(e) = avg_loss_removed(e) - avg_loss_kept(e)
     or use regression coefficient βe.

  5. Protect structurally critical PIPs:
     do not remove if it breaks minimum BEL input reachability,
     output escape, straight/turn routing, carry/direct paths, etc.

  6. Sort by:
     remove_score = cost_saving / importance

  7. Remove PIPs in small batches.
     After each accepted batch, rerun full demand tester.

  8. Stop when:
     hard demands fail,
     soft failure exceeds threshold,
     or timing/path quality gets too bad.
  We should then also store a matrix that shows each importance that we
  calculated for each pip. Similar to the switch matrix csv but the
  values are the importance values.
